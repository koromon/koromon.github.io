JavaScript DOM事件

发表于 2018-01-15

事件

理解事件流

事件流描述页面接收事件的顺序

  • 事件冒泡流（IE和现代浏览器），由最具体的元素传播到不具体的节点。建议使用
  • 事件捕获流（现代浏览器），由不具体的节点到具体的节点

全部浏览器都支持事件冒泡。

DOM2级事件规定的事件流包括三个阶段：事件捕获阶段、处于目标阶段和事件冒泡阶段。

Tips: IE9及以上才全部实现 DOM2级 事件（事件流）。

使用事件处理程序

事件就是用户或者浏览器执行的某种动作，而响应某个事件的函数就叫做事件处理程序。

  • HTML 事件处理程序
    • 就是在 HTML 直接写 JS，基本很少使用，耦合性太强
  • DOM0 级事件处理程序
    • 获取元素后直接对元素 ele.onxx 进行函数赋值，可以通过 ele.onxxx = null; 删除事件，全部浏览器都支持，但每种事件只能绑定一种方法。
    • 绑定的属性方法名要小写
    • 事件处理程序会在冒泡阶段被处理
    • this 指向元素的作用域
  • DOM2 级事件处理程序
    • 接受三个参数（事件名/处理函数/布尔值 true为捕获，false为冒泡）
      • addEventListener()
      • removeEventListener()(无法移除匿名函数)
    • 可以添加多个事件处理程序，先添加的先执行。
    • this 指的是元素作用域
  • IE 事件处理程序
    • IE9 之前只支持事件冒泡，所以 IE 事件处理程序是在冒泡阶段执行。只接受两个参数（on + 事件名/处理函数）
      • attachEvent()
      • detachEvent()
    • 可以添加多个事件处理程序，后添加的先执行。
    • this 指的是 window

Tips: 在事件处理程序内部，对象 this 始终等于 currentTarget 的值，而 target 只包含事件的实际目标。也就是说直接点击目标元素的时候，这三个值相等；如果点击父节点（比如 body）触发事件程序，此时 this 等于 currentTarget，而 target 元素指向真正的目标（比如 body 上的按钮）。

IE9 及以上/现代浏览器都支持 DOM 事件操作。

  • IE 中的事件对象（IE9 以下）
    • DOM0 通过 window 可以访问 event，直接获取参数的 event 会是 undefined
    • attachEvent() 添加的事件对象(event)可以通过传入的参数直接获取也可以通过 window 获取
    • 事件的属性和方法

事件类型

  • DOM3 级别，重新定义了 DOM2 中的一些事件，也新增了一些事件。(了解常用的即可，其他可以上网查询，具体内容可以去 w3c 查看)

IE9 开始支持 DOM3 事件。

*内存和性能

  • 事件委托

    • 利用事件冒泡，假设有多个元素要添加事件处理程序，那么可给他们的一个父类 DOM 元素添加事件处理程序（节省资源），再根据实际 target 对子类元素进行处理。
    • 比较适合采用事件委托的事件（click/mousedown/mouseup/keydown/keyup/keypress）（这些事件都会冒泡，而且要有动作触发）

  • 移除事件和处理程序

    • 对于直接删除或者替代掉的 DOM 元素，要在移除之前先移除他们绑定的事件（remove/null）
    • 通过 onunload 事件，统一移除所有事件处理程序。

Tips:如果使用了事件委托，就没那么麻烦，因为全部事件都只绑定在一个 DOM 上。

自定义事件

  • 普通自定义事件（无法传参）
  • CustomEvent 可以传递对象参数

CustomEvent() 创建自定义事件(现代浏览器基本支持)，合理使用可以减少 DOM 和事件之间的耦合性，提高代码复用效率，具体用法可以参考 MDN。

小结

  • 从性能方面考虑，不要滥用事件处理程序。
  • 可以多用事件委托技术。
  • 浏览器卸载页面之前移除所有事件处理程序（目前来说这个应该不需要了，浏览器应该已经有处理了）
  • 可以使用 js 模拟事件（这个要用的时候再搜索一下就足够了，目前先了解一下）

小结

之前已经了解到，React.js 可以通过 context 设置‘全局变量’。这样父组件下的子组件就可以任意获取 context 中的变量，而不用一层层的进行状态提升。但是这样每个子组件都可以修改 context 会导致程序运行的不可预料。而在前一节理解 Redux中，我们了解到 store 是不会被随意修改的。于是我们考虑 context 和 store 进行结合。

在创建了 store 后，我们把它放到组件的 context 中。然后在子组件中获取 store，再进行和 Redux 类似的操作即可。

在这个过程中，我们发现每个子组件中会出现很多重复获取 context 的代码。此时我们可以实现一个高阶组件 connect，这个高阶组件获取 mapStateToProps（设置子组件需要的属性）、mapDispatchToProps（设置子组件需要的方法）以及 WrappedComponent（被包裹的子组件）三个参数，再把通过这前两个函数得到的内容通过 props 传回去给子组件。

最后我们再设置一个根组件（Provider），这个组件提供了 context 中的 sotre。

通过这样的设置，我们的组件都能抽离为 Dump 和 Smart 组件。在 Dump 组件中，我们不需要处理数据，只需要根据 Smart 组件中传入的 props 进行显示即可。所以这样的 Dump 组件是可以复用的。 而 Smart 组件则是用来执行特定应用逻辑的，不用考虑太多复用性问题。

CSS盒模型

发表于 2018-01-09

盒模型

盒模型指定元素如何显示以及（在某种程度上）如何交互。页面上的每个元素被看做一个矩形框（也称为元素框），这个框由元素的内容（Content）、内边距（Padding）、边框（Border）和外边距（Margin）组成。示意图如下：

在盒模型中，外边距可以为负值，但是内边距和边框不能为负值。而盒模型又分为两种模型。

标准模型

在标准模型中，CSS 中的 width 和 height 指的是内容区域的宽度和高度。增加内边距、边框和外边距不会影响内容区域的尺寸，但是会增加元素框的总尺寸。示意图如下：

IE 模型

但是在 IE 盒模型（IE 早期浏览器）中，CSS 中的 width 和 height 指的是内容区域、内边距和变宽的宽度总和。示意图如下：

设置盒模型

CSS3 中 box-sizing 属性可以定义使用哪种盒模型。

box-sizing:content-box; 使用标准模型

box-sizing:border-box; 使用 IE 模型

JS 如何获取盒模型的宽高

  • 对于内联样式，可以通过： dom.style.width/height
  • 渲染后的节点样式，可以通过：dom.currentStyle.width/height（只有IE支持）
  • 渲染后的节点式，可以通过： window.getComputedStyle(dom).width/height （高级浏览器）
  • 渲染后的节点式，可以通过： dom.getBoundingClientRect().width/height （返回一个对象，包含元素宽/高，以及相对于视口左上角的 top/left/right/bottom）

外边距叠加

当两个或更多垂直外边距相遇的时候，它们将形成一个外边距。这个外边距的高度等于两个发生叠加的外边距的高度中的较大者。

  • 常见的外边距叠加情形
    • 上下两个元素，第一个元素的底部外边距和第二个元素的顶部外边距叠加
    • 当一个元素包含在另一个元素中（另一个元素没有设置内边距或者边框的时候）
    • 一个空元素（没有设置内边距或者边框的时候），它本身的顶部和底部边距也会发生叠加。如果这个外边距继续和其他元素的外边距相遇，也会发生叠加。

块级格式化上下文（Block Formatting Contexts）

从样式上看，具有 BFC 的元素与普通的容器没有什么区别，但是从功能上，具有 BFC 的元素可以看作是隔离了的独立容器，他有一些普通容器没有的特性：

  • 独立的容器，不会和外部容器相互影响（内部元素不会在布局上影响到外面的元素）
  • 垂直方向不会发生重叠
  • 不会与浮动元素重叠
  • 可以包含浮动元素（浮动元素也参与高度计算）。

所以当我们不想发生边距重叠或者不希望和浮动元素重叠的时候，我们可以创建 BFC 元素。创建 BFC 元素有下面四种方法：

  • 浮动元素：float 除了 none 以外的值
  • 定位元素 position 除了 static/relative 以外的值
  • display 为 inline-block/table-cell/table-caption
  • overflow 除了 visible 以外的值（hidden，auto，scroll）

因为 .right 创建了 BFC，所以其父元素的宽度等于 .right 的水平属性值之和，即外边距 + 内边距 + 边框宽度 + 宽度 = 父元素宽度。所以当 .right 的负外边距大于 150px 的时候，其父元素的宽度为 0，也就会挤上去第一排。

常见三栏布局的实现

发表于 2018-01-05

实现一个三栏布局，高度已知，中间宽度自适应，左右定宽的页面布局。

这里使用了五种方式来解决上面的问题。分别是下面几种：

一、 浮动定位

浮动定位兼容性好，但是设置这种布局需要注意调整 html 中元素的位置。同时因为设置元素浮动会导致脱离文档流，所以必要的时候需要要清除浮动。

清除浮动有几种常见的方法

  • 在浮动元素的下方新增一个标签（block/inline-block 类型），然后该标签样式中使用 clear:both，因为要增加额外的元素，不推荐使用
  • 浮动元素的父元素设置 overflow:auto;（这个可能会产生滚动条） 或者 overflow:hidden;（这个会导致超出的元素隐藏）

  • 使用伪元素,只要给浮动元素的父元素添加相应的类名即可

  • 双伪元素，和上面的类似，但是写的属性不用那么多。推荐使用

二、 绝对定位

绝对定位不需要特意调整 html 中的元素结构，但是同样会导致元素脱离文档流。需要注意对布局的影响。

三、 Flexbox

CSS3 中新增的布局样式，在目前来说，兼容性已经比较好了，尤其是移动端基本算是全部机型兼容了。Flexbox 布局很方便，要熟悉常见的属性，有助于提升开发效率。用法可以参考这里 - 理解Flexbox：你需要知道的一切。

四、 table 布局

兼容性很好，而且它有个特性，就是强制每行中的每列都会等高，但是我再实际开发中比较少用到这种布局。在某些情况下这种布局也是一种很好解决方案。table布局的简单介绍

五、 Grid 布局

Grid 作为一种新的布局方式，兼容性上还不是很好，但是它把页面整体看做一个网格，使得我们可以随意摆放元素的位置。这种布局我在开发中还没用过，不过了解一下还是不错的。Grid 布局的简单介绍

综上所述，从兼容性和便利性的角度来说，Flexbox 是一种比较完美的布局方式。

接下来我们改一下布局要求：实现一个三栏布局，高度不定，中间宽度自适应，左右定宽的页面布局。

一、 浮动定位，此时父元素的高度只会根据中间元素的高度固定，而浮动的左右两个元素因为脱离的文档流，无法获取父元素高度。此时两边的高度无法跟随父元素。

二、 绝对定位，此时三个元素都脱离文档流，父元素无法获取高度。无法实现

三、 Flexbox，左右两列会随着中间的高度变化而变化，可以实现。

四、 table 布局，因为 table 会强制同行的每列等高，所以可以实现。

五、 Grid 布局也可以实现。

《React.js 小书》学习 - 理解 Redux

发表于 2018-01-04

React.js 小书原系列地址

高阶组件

高阶组件是一个函数，传给它一个组件，返回一个新的组件（你传给它的组件作为子组件）。

其实这就是为了组件之前的代码复用。把组件之间相同的逻辑抽离出来，放到高阶组件中进行复用。高阶组件内部的包装组件和被包装组件之间通过 props 传递数据。

context

组件树上某颗子树的全局变量，它的子组件可以访问这个变量，父组件无法访问。

因为 context 的内容可以随意被接触也就可以随意被修改，所以很不安全。

这个特性强烈建议不使用。建议使用第三方库管理组件状态

Redux

一种前端“架构模式”。

开发中，要尽量避免全局变量，否则每个模块都可以对共享数据进行操作，会导致无法预料的错误。但是不同的模块之间确实又需要共享数据。

在 React.js 中。我们设置一个专门的函数负责数据的操作，所有的数据操作都要通过这个函数，而这个函数决定可以进行什么修改。这样在 debug 的时候我们可以立刻找出什么地方修改了共享数据。

当监听到数据变化的时候，这个函数内部也应该主动重新渲染页面（观察者模式）。

纯函数

一个函数的返回结果只依赖于它的参数（不受外部变量的影响），并且在执行过程中没有副作用（不会导致外部变量的改变）。同时会导致外部可观察的变化（页面刷新，DOM 变化，console.log），都属于副作用。

所以纯函数就是只涉及计算数据，不会产生不可预料的行为，也不会影响外部。对于代码开发和维护非常友善。

共享结构的对象如何提升性能？

每次更改状态的时候判断是哪些部分的数据发生改变，对应的只需要重新渲染该部分的组件即可。

reducer

其实这个就是一个纯函数，不会影响外部环境。

如何创建一个 Redux

首先，为了方便管理，我们统一创建一个函数（dispatch）用于管理数据，数据的更改要通过 action 声明，这样便于管理和调试。然后我们可以把获取初始数据的方法（getState）和管理函数（dispatch）合并到一个函数中（抽象出一个类，可以产生实例） 。

有了统一管理数据的函数，但是此时更新数据后我们还需要手动渲染页面，所以我们要在 dispatch 中监听数据变化，一旦发生变化就重新渲染组件。

为了使得这种监听方式更加通用，我们采取观察者模式。同时使得每个实例都有独立的监听函数。

此时如果发生数据变动已经可以自动渲染了，但是只有一小部分数据改变的时候，全部组件都会进行渲染（性能问题），所以我们需要判断哪些数据发生了变动，只需要渲染对应部分即可。

我们可以通过判断前后两次数据的内容来确认哪部分的数据发生了变化，这里需要通过浅复制的操作（因为是对象）来实现。每次发生数据变化，都要生成一个新的对象。渲染的时候如果前后两个对象相同，即证明没有发生变化，也就不用重新渲染这部分。

接下来我们把初始化数据的部分也合并到管理数据的函数中。此时我们发现这个合并后的函数就是一个纯函数 reducer，它不会改变外部的数据，只会初始化和计算新的 state。

最后，我们已经创建了一个独立的函数（类），只要传入不同的 reducer，就可以生成不同的 Redux。

一个完整的流程： 定一个 reducer -> 生成 store -> 监听数据变化重新渲染页面 -> 首次渲染页面 -> 后面可以随意 dispatch 了，页面自动更新

小结

由此可见，Redux 这种模式，允许我们通过统一的方式管理我们的应用。先预先设定动作（action），然后把动作传入函数（reducer），函数内部根据传入的动作（action）改变原来的状态（state）。

状态管理 - 状态提升

把组件之间共享的状态交给组件最近的公共父节点保管，然后通过 props 把状态传递给子组件，实现组件数据共享。把子孙组件的状态传递给父组件，也称为状态提升。

当某个状态被多个组件依赖或者影响的时候，就应该使用状态提升。后面可以使用 Redux 进行状态管理。

挂载阶段的组件生命周期

React.js 将组件渲染，并且构造 DOM 元素然后塞入页面的过程称为组件的挂载。

生命周期的应用

constructor 中一般进行组件的 state 初始化

componentWillMount 中一般进行组件的启动动作，比如 Ajax 数据拉取，定时器的启动等。

componentDidMount 中一般进行对于 DOM 有依赖的操作。

componentWillUnmount 中可以对一些定时器进行销毁或者数据清理。

更新阶段的组件生命周期

使用 setState 导致重新渲染组件的过程，这是一个组件变化的过程，也有一系列生命周期函数。

  1. shouldComponentUpdate(nextProps, nextState)：你可以通过这个方法控制组件是否重新渲染。如果返回 false 组件就不会重新渲染。这个生命周期在 React.js 性能优化上非常有用。
  2. componentWillReceiveProps(nextProps)：组件从父组件接收到新的 props 之前调用。
  3. componentWillUpdate()：组件开始重新渲染之前调用。
  4. componentDidUpdate()：组件重新渲染并且把更改变更到真实的 DOM 以后调用。

ref 和 React.js 中的 DOM 操作

在 JSX 中给某个元素添加 ref 属性，ref 接受一个回调函数，在加载的时候把 DOM 元素传入 ref 的回调函数中。

要尽量减少 ref 的使用。关于 ref 的官方文档

props.children

使用自定义组件的时候，嵌套在组件内部的结构都可以通过 props.children 获取，获取的这个内容是一个数组。JSX 会自动把它们渲染出来。

dangerouslySetHTML 和 style 属性

React.js 中所有表达式插入的内容都会自动转义，类似 jQuery 中的 text(..) 函数。只能通过 dangerouslySetHTML 属性动态设置元素的 innerHTML，要传入 __html 属性值。

这个属性可能会导致 XSS 攻击，要慎用。

React.js 中的 style 属性要传入一个对象，并且只能使用驼峰命名法。

PropTypes 和组件参数验证

因为 js 弱类型的特性，为了保证代码的健壮性，需要设置参数验证。使得组件开发更加规范。PropTypes 官方文档

代码规范建议

组件的私有方法使用 _ 开头，所有事件监听的方法使用 handle 开头。把事件监听方法传给组件的时候，属性名用 on 开头。

组件的内容编写顺序如下：

  1. static 开头的类属性，如 defaultProps、propTypes。
  2. 构造函数，constructor。
  3. getter/setter。
  4. 组件生命周期。
  5. _ 开头的私有方法。
  6. 事件监听方法，handle*。
  7. render* 开头的方法，有时候 render() 方法里面的内容会分开到不同函数里面进行，这些函数都以 render* 开头。
  8. render() 方法。

《React.js 小书》学习 - 基本用法

React.js 小书原系列地址

React.js 解决什么问题？

解决前端结构的复用性问题，抽离出独立的类，更好的完成组件化。

减少手动的 DOM 操作。一旦状态发生变化，就自动构建一个新的 DOM 元素并替换原先的 DOM 元素。

React 中通过 Virtual-DOM 减少重排消耗。

概括：一个组件有自己的显示形态（上面的 HTML 结构和内容）行为，组件的显示形态和行为可以由数据状态（state）和配置参数（props）共同决定。数据状态和配置参数的改变都会影响到这个组件的显示形态。当数据变化的时候，组件的显示需要更新。

React.js 项目的基本结构

import React, { Component } from 'react' - 这两个是必须要引用的

import ReactDOM from 'react-dom' - 负责把描述 UI 信息的 JavaScript 对象变为 DOM 对象， 并渲染到页面上

JSX 其实就是 JavaScript 对象，所有的 DOM 元素其实都可以用 JavaScript 对象来表示。

JSX 到页面经过了下面的过程：

组件的 render 方法

一个组件类必须实现一个 render 方法，而这个 render 方法必须返回一个 JSX 元素（只能有一个父元素）

render 方法中 {} 里面可以放任何的 js 代码。

JSX 中的类名 class 要用 className 代替。for 属性要用 htmlFor 代替。因为它们都是关键字。

自定义的组件都必须要用大写字母开头，普通的 HTML 标签都用小写字母开头。

事件监听

直接给元素加上类似 onClick 这样的属性（驼峰命名法），然后再绑定方法即可。on* 的事件监听只能作用在普通的 HTML 标签上。

事件监听函数会被自动传入 event 对象。

为了获取事件中的 this，需要手动的将实例方法绑定（bind）到当前实例。一个事件中如果执行多个函数，需要用一个函数套起来，或者用逗号隔开。

组件的 state 和 setState

state 用来存储可以变化的状态。

setState 接受对象参数，这个方法是父类 Component 提供的，调用的时候，React.js 会更新组件的状态 state，并且重新调用 render 方法，再把结果显示到页面上。传入对象的时候，可以只传入要变更的属性。

setState 接受函数参数。因为调用 setState 的时候，React.js 并不会马上修改 state。而是存入更新队列，最后再触发更新。如果传入函数作为参数，React.js 会把上一个 setState 的结果传入这个函数，然后返回一个更新的 state 对象。

注意：更改组件的状态时，一定要用 React.js 提供的 setState 方法，否则 React.js 不知道你修改了组件的状态。

React 会对 setState 进行优化，即使调用多次 setState，也只会渲染一次组件。

配置组件的 props

每个组件都可以接受一个 props 参数，作为对组件的配置。

使用组件的时候，可以把参数放在标签的属性当中，所有的属性都会作为 props 对象的键值。

字符串、数字、对象、数组、函数等，都可以作为 props 传入组件

默认配置 defaultProps，这个对象作为默认的 props 对象。

不能在组件内部更改传入的 props，但是可以主动地重新渲染新的方式传入新的 props（父组件中重新渲染）。

state 和 props

state 只可以在组件内部通过 setState 更新，会导致组件重新渲染。
props 只能通过外部更改。

state 是让组件控制自己的状态，props 是让外部对组件自己进行配置。

尽量少用 state（更多的状态只会导致更难的管理），多用 props（高配置性）

渲染列表数据

如果在 {} 中放一个数组，React.js 会帮你把数组里面的元素一个个罗列并渲染出来。此时要为每个元素加上 key 属性作为唯一标识。

处理用户输入

React.js 中所有的状态都应该由 state 控制，如果输入控件（<input /> 等）设置了值，那么值不变，value 就不会变。所以要通过 setState 更新组件的内容。这些值被 React 控制的组件叫做受控组件。

向父组件传递数据

通过 props 设置回调函数，可以将子元素的数据传到父元素中。

按照 layout 设置的默认模板新建一篇文章。

启动服务器。默认情况下，访问网址为： http://localhost:4000/。这样就可以在本地看到部署后的效果了。

$ hexo generate

生成静态文件，也就是生成最新的资源，一般在部署最新文章前都要先生成资源。而这个指令有两个选项

  • -w, --watch 监视文件变动，也就是本地修改文章后会实时生成最新的资源

$ hexo deploy(简写 $ hexo d)

把网站部署到线上。这个指令有一个选项

  • -g, --generate 部署之前预先生成静态文件，如果执行这个选项就可以省略上面上面生成资源那个指令。
